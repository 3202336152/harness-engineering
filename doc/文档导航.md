# 文档导航

这个目录的目标不是“堆更多文档”，而是把不同用途的文档分开，避免把当前实现、操作说明、方法论、历史草稿混在一起。

| 你想解决的问题 | 建议先看 | 说明 |

| 我只想看 `/harness init`、`/harness audit`、`/harness plan` 怎么用 | [./命令使用说明.md](./命令使用说明.md) | 三个核心命令的适用场景、参数、产物、示例 |
| 我想看能力边界和功能介绍 | [./能力与功能说明.md](./能力与功能说明.md) | 当前支持什么、产出什么、边界在哪里 |

## 当前文档分层

| 分层 | 路径 | 定位 | 是否要求和当前实现同步 |
|------|------|------|------------------------|
| 当前真相 | `README.md`、`SKILL.md`、`doc/本地使用指南.md`、`doc/命令使用说明.md`、`doc/能力与功能说明.md`、`doc/安装与试点指南.md` | 当前可执行能力、用法、维护说明 | 是 |
| 理念手册 | `doc/手册/` | 方法论、实践建议、推广思路 | 部分同步，允许比实现更泛化 |
| 历史设计 | `doc/历史设计/` | 设计过程记录 | 否 |
| 归档草稿 | `doc/归档/` | 旧方案留档 | 否 |

## 文档维护规则

- 修改命令行为、脚本能力、模板能力时，至少同步：
  - `README.md`
  - `SKILL.md`
  - `doc/本地使用指南.md`
  - `doc/命令使用说明.md`
  - `doc/能力与功能说明.md`
  - `doc/安装与试点指南.md`
- 修改 `doc/` 目录结构、文档职责、阅读入口时，至少同步：
  - `doc/文档导航.md`
- 补充方法论、推广策略、经验总结时，优先更新：
  - `doc/手册/`
  - `references/`
- `doc/` 下的维护文档继续使用中文文件名，便于团队浏览和比对。
- 项目级共享规范统一落在 `docs/project/`，功能级规范统一落在 `docs/features/`。
- 项目级与功能级 spec 默认用中文内容和中文 `md` 文件名；脚本通过 `manifest.json` 与 `.harness/*.json` 识别，不再依赖英文文件名。
- `doc/历史设计/` 和 `doc/归档/` 默认不要求与当前实现逐条同步。
- 如果后续还要在 `doc/` 顶层新增文档，先判断它属于：
  - 当前维护文档
  - 理念手册
  - 历史资料
  尽量放到对应分层，避免顶层继续膨胀。
