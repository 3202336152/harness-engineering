# 本地使用指南

这份文档是给当前仓库维护者、试点推广负责人、以及准备在业务项目里落地 Harness Engineering 的同学看的。

它重点回答四个问题：

1. 这个仓库到底是干什么的。
2. 在业务项目里应该怎么用。
3. 模板、规范、文档分别应该放在哪里。
4. 模板后续升级或个性化定制时怎么管理。

> 如果你只想看仓库对外简介，先看 [README.md](../README.md)。
> 如果你只想看 Skill 契约和命令入口，先看 [SKILL.md](../SKILL.md)。
> 如果你想看方法论，不是操作步骤，去看 [doc/手册/Harness工程手册.md](./手册/Harness工程手册.md)。

## 一句话说明这个仓库

这个仓库不是业务项目本身，而是一个“Skill + 模板 + 脚本”的标准化工具包。

它的目标是帮助业务项目统一以下内容：

- AI 代理入口文档，例如 `AGENTS.md`、`CLAUDE.md`
- 项目级共享规范，例如架构、设计、接口、开发、测试、安全、需求
- 功能级 spec，例如功能概览、设计、接口、数据库、测试、发布、状态
- 模板覆盖机制，让团队模板、项目模板、个人模板可以共存
- 校验与治理机制，让 spec 不只是“生成出来”，而是能长期维护

## 先分清 4 层东西

这是最容易混淆的地方，先分清之后后面的内容就好理解了。

| 层次 | 位置 | 作用 | 谁维护 |
|------|------|------|--------|
| 仓库维护文档 | `doc/` | 解释当前仓库怎么维护、怎么使用、为什么这么设计 | 当前 Skill 仓库维护者 |
| Skill 契约 | `SKILL.md` | 定义 Skill 能力、命令入口、引用的参考资料 | 当前 Skill 仓库维护者 |
| 内置模板与脚本 | `assets/templates/`、`scripts/` | 生成项目文档、执行校验、提供治理能力 | 当前 Skill 仓库维护者 |
| 业务项目落地产物 | 业务项目里的 `docs/`、`.harness/` | 业务项目真实使用的规范、spec、治理配置 | 各业务项目团队 |

一句话理解：

- `doc/` 解释“这个 Skill 仓库自己怎么维护”。
- `assets/templates/` 是“生成业务项目文档的母版”。
- 业务项目里的 `docs/` 和 `.harness/` 才是团队真正每天会看的东西。

## `doc/` 和 `docs/` 的区别

这个问题一定要先说清楚。

| 路径 | 含义 | 是否当前业务项目真相 |
|------|------|----------------------|
| `doc/` | 当前 Skill 仓库自己的维护文档 | 否 |
| `docs/project/` | 某个业务项目的项目级共享规范 | 是 |
| `docs/features/` | 某个业务项目的功能级 spec | 是 |
| `docs/exec-plans/` | 某个业务项目的执行计划 | 是 |

所以：

- 你在这个仓库里修改 `doc/本地使用指南.md`，是在维护这个 Skill。
- 你在业务项目里修改 `docs/project/ARCHITECTURE.md`，是在维护业务项目的真实规范。

## 一张图看懂完整工作流

```text
当前仓库（Skill 仓库）
  ├─ scripts/                  # 命令实现
  ├─ assets/templates/         # 内置模板
  ├─ SKILL.md                  # Skill 契约
  └─ doc/                      # 维护文档

业务项目（Skill 落地后）
  ├─ AGENTS.md / CLAUDE.md
  ├─ docs/project/*.md         # 项目级共享规范
  ├─ docs/features/<feature>/  # 功能级 spec
  ├─ docs/exec-plans/          # 执行计划
  ├─ .harness/spec-policy.json # 规则与模板元数据
  └─ .harness/templates/       # 项目级覆写模板（可选）
```

更通俗一点的流程是：

1. 维护者在这个仓库里维护模板和脚本。
2. 业务项目执行 `init`，生成第一版 Harness 基础结构。
3. 项目负责人补全 `docs/project/` 下的项目级共享规范。
4. 开发某个功能时，先生成 `docs/features/<feature>/` 下的功能级 spec。
5. 如果有人工代码改动，先用 `check-doc-impact` 判断本次代码是否要求同步更新文档。
6. 开发完成后，用 `validate-spec` 做结构校验，用 `--strict` 做内容质量门禁。
7. 模板后续如果升级，用 `check-template-drift` 检查项目文档和模板是否脱节。

## 初始化后会生成什么

运行 `bash scripts/init-harness.sh ...` 之后，业务项目里通常会出现下面这些东西。

| 路径 | 用途 | 是否建议手工编辑 |
|------|------|------------------|
| `AGENTS.md` | 面向 AI 代理的入口索引 | 是 |
| `CLAUDE.md` | 面向 Claude Code 的兼容入口 | 是 |
| `docs/project/ARCHITECTURE.md` | 项目架构和边界 | 是 |
| `docs/project/DESIGN.md` | 项目设计原则和 Java 分层设计约定 | 是 |
| `docs/project/API-SPEC.md` | 项目接口协议总规范 | 是 |
| `docs/project/DEVELOPMENT.md` | 开发规范、编码规则、评审要求 | 是 |
| `docs/project/REQUIREMENTS.md` | 项目背景、需求、验收口径 | 是 |
| `docs/project/TESTING.md` | 测试分层、门禁、回归口径 | 是 |
| `docs/project/SECURITY.md` | 安全约束、输入校验、审计要求 | 是 |
| `docs/features/` | 功能级 spec 根目录 | 是 |
| `docs/exec-plans/` | 执行计划目录 | 是 |
| `.harness/architecture.json` | 架构边界配置 | 是 |
| `.harness/spec-policy.json` | 项目级/功能级 spec 规则、模板元数据 | 是 |
| `.harness/doc-impact-rules.json` | 代码改动与文档更新的影响规则 | 是 |
| `.github/PULL_REQUEST_TEMPLATE.md` | PR 模板 | 是 |

另外还会保留兼容入口：

- `docs/ARCHITECTURE.md`
- `docs/CONVENTIONS.md`
- `docs/TESTING.md`
- `docs/SECURITY.md`

这些兼容入口主要是为了兼容部分工具或旧流程，当前更推荐把项目级共享规范放在 `docs/project/`。

## 命令与场景说明

这一节尽量用“什么时候用、输入什么、会产出什么”的方式来讲。

### 1. `init-harness.sh`

命令：

```bash
bash scripts/init-harness.sh [--project-name <name>] [--description <text>] [--template-root <path>] [--profile <name>] [--force] [--dry-run]
```

什么时候用：

- 新项目第一次接入 Harness
- 老项目补齐基础结构
- 想预览初始化会生成什么

它会做什么：

- 检测当前目录是不是 Git 仓库
- 检测技术栈
- 生成 `AGENTS.md`、`CLAUDE.md`
- 生成 `docs/project/` 下的项目级共享规范
- 生成 `.harness/spec-policy.json`
- 生成 `.harness/doc-impact-rules.json`
- 生成执行计划目录和 PR 模板

技术栈识别规则：

- 有 `package.json`，识别为 `node`
- 有 `pom.xml`，识别为 `java-maven`
- 有 `build.gradle` 或 `build.gradle.kts`，识别为 `java-gradle`
- 有 `pyproject.toml` 或 `setup.py`，识别为 `python`
- 有 `go.mod`，识别为 `go`
- 有 `Cargo.toml`，识别为 `rust`
- 其他情况为 `unknown`

常用参数说明：

- `--project-name`：指定项目名
- `--description`：指定项目描述
- `--profile`：覆盖默认模板画像
- `--template-root`：单次执行使用自定义模板根目录
- `--force`：覆盖已存在文件
- `--dry-run`：只预览，不真正写文件

最常用示例：

```bash
bash scripts/init-harness.sh --project-name order-center --description "订单中心"
```

如果你只是想先看会生成什么：

```bash
bash scripts/init-harness.sh --project-name order-center --dry-run
```

对 Java 项目来说，最重要的一点是：

- Maven 项目默认 profile 是 `java-backend-service`
- 也可以手工指定为 `java-batch-job`
- 或者指定为 `java-adapter`

### 2. `audit-harness.sh`

命令：

```bash
bash scripts/audit-harness.sh
```

什么时候用：

- 想知道一个项目的 Harness 成熟度
- 想评估老项目目前差在哪
- 想知道优先应该补哪些内容

它会检查什么：

- 入口文档
- 文档结构
- 文档新鲜度
- 架构约束
- 测试准备度
- 自动化
- 执行计划
- 安全治理

它的输出是结构化 JSON，最值得看的字段通常有：

- `overall_score`
- `maturity_level`
- `maturity_label`
- `priority_fixes`

简单理解：

- `overall_score` 越高，说明项目对 AI 代理越友好
- `priority_fixes` 是最适合先补的东西

如果你是第一次评估一个老项目，通常建议先跑这个命令。

### 3. `plan-harness.sh`

命令：

```bash
bash scripts/plan-harness.sh --task "<任务描述>" [--agent <name>] [--output-dir <path>] [--dry-run]
```

什么时候用：

- 任务比较复杂，需要先拆计划再开发
- 想让代理先基于项目级规范生成执行计划

默认产物：

- `docs/exec-plans/active/<slug>.md`

示例：

```bash
bash scripts/plan-harness.sh --task "新增订单查询能力" --agent codex
```

这个命令不会实现功能，它只负责把任务整理成可执行的计划，并明确约束、验收标准和阶段步骤。

### 4. `new-feature-spec.sh`

命令：

```bash
bash scripts/new-feature-spec.sh --id <feature-id> --title <title> [--owner <name>] [--change-types <csv>] [--config <path>] [--template-root <path>] [--dry-run]
```

什么时候用：

- 新功能开始前，需要先把功能级 spec 建出来
- 想按统一模板给某个功能生成设计文档、接口文档、测试文档等

默认一定会生成的文档：

- `overview.md`
- `design.md`
- `test-spec.md`
- `status.md`

默认 `change_types` 扩展规则：

- `api`：额外生成 `api-spec.md`
- `db`：额外生成 `db-spec.md`
- `rollout`：额外生成 `rollout.md`

示例：

```bash
bash scripts/new-feature-spec.sh --id FEAT-001 --title "订单查询" --owner alice --change-types api,db
```

这个命令会在下面目录下生成一组文档：

```text
docs/features/FEAT-001-订单查询/
```

要注意两点：

- 目录名支持中文标题 slug
- 文档内容默认是中文，但路径保持稳定命名，便于脚本和 CI 识别

### 5. `validate-spec.sh`

命令：

```bash
bash scripts/validate-spec.sh --json
bash scripts/validate-spec.sh --json --strict
```

什么时候用：

- 想检查项目级 spec 和功能级 spec 是否齐全
- 想在 CI 中增加 spec 门禁

普通模式会做什么：

- 检查项目级必需文档是否存在
- 检查功能级必需文档是否存在
- 检查功能目录是否完整

严格模式会额外检查什么：

- 必填章节是否存在
- 必填 frontmatter 是否存在
- `template_version/profile/language` 是否与 `.harness/spec-policy.json` 一致
- 是否还保留明显占位文本，例如 `TODO:`、`待补充`

为什么新生成的文档通常会在 `--strict` 下失败：

- 因为模板里本来就保留了待填写内容
- 这是一种有意设计，目的是让“脚手架成功”和“内容合格”分成两个阶段

推荐做法：

1. 先用普通模式确认结构已经生成完整。
2. 填完关键内容后，再跑 `--strict`。
3. 团队成熟后，把 `--strict` 接到 CI。

### 6. `check-doc-impact.sh`

命令：

```bash
bash scripts/check-doc-impact.sh --json --staged
bash scripts/check-doc-impact.sh --json --base-ref <sha> --head-ref <sha>
```

什么时候用：

- 担心人工改了代码，但没同步更新文档
- 想在本地提交前做一次轻量门禁
- 想在 PR / MR 的 CI 中做强制校验

它检查的不是“AI 看没看到”，而是“git diff 里发生了什么变化”。

默认内置的判断思路是：

- 改了 Controller / OpenAPI / client，应该更新接口文档
- 改了 DB migration / SQL / Mapper，应该更新数据库文档
- 改了 security / auth 相关代码，应该更新安全文档
- 改了构建、配置、部署文件，应该更新开发或发布文档

本地最常用方式：

```bash
bash scripts/check-doc-impact.sh --json --staged
```

CI 最常用方式：

```bash
bash scripts/check-doc-impact.sh --json --base-ref <base-sha> --head-ref <head-sha>
```

规则文件位置：

```text
.harness/doc-impact-rules.json
```

如果团队想扩展更多判断规则，例如缓存、消息、搜索索引、SLA、风控等，可以直接改这个文件。
如果你想快速落地本地提交门禁，可以参考：

```text
assets/hooks/pre-commit-doc-guard.sh.tpl
```

### 7. `prepare-template-overrides.sh`

命令：

```bash
bash scripts/prepare-template-overrides.sh --list
bash scripts/prepare-template-overrides.sh --template feature/overview.md.tpl
bash scripts/prepare-template-overrides.sh --output-root .harness/templates --force
```

什么时候用：

- 想知道当前有哪些模板可以改
- 想把内置模板导出一份到项目里做定制
- 想批量导出模板做团队共用修改

常用方式：

先看模板列表：

```bash
bash scripts/prepare-template-overrides.sh --list
```

再导出某一个模板：

```bash
bash scripts/prepare-template-overrides.sh --template feature/overview.md.tpl
```

导出以后，文件会出现在：

```text
.harness/templates/feature/overview.md.tpl
```

后续再执行 `new-feature-spec.sh` 时，这个项目就会优先使用这个覆写模板。

### 8. `check-template-drift.sh`

命令：

```bash
bash scripts/check-template-drift.sh --json
```

什么时候用：

- 模板升级之后，想知道已有文档是不是旧了
- 团队里已经有人开始定制模板，想知道当前 override 有没有混乱
- 想准备部门推广前，先做一次模板治理审计

它会检查两类问题。

第一类：文档元数据漂移

- 项目级和功能级文档里的 `template_version`
- 文档里的 `template_profile`
- 文档里的 `template_language`

第二类：模板覆写治理

- 冗余 override：导出后完全没改，和内置模板一模一样
- 自定义 override：和内置模板不同，是真实的团队定制
- 孤儿模板：已经无法映射到当前内置模板，可能是历史遗留物

这条命令很重要，因为它回答的是“模板升级后，项目还有多少历史债”。

### 9. 其他辅助命令

`lint-architecture.sh`

- 检查架构边界是否被破坏
- 适合接到 CI

`check-doc-freshness.sh`

- 检查 `docs/` 下文档是否长时间没更新
- 适合做文档健康度提醒

`verify-spec-compliance.sh`

- 检查当前 Skill 仓库自身结构是否符合要求
- 更多是给当前仓库维护者用

`publish-check.sh`

- 串联本地测试和发布前校验
- `--skip-official` 适合本地先跑一遍

## Java 项目推荐落地流程

如果你们部门大多数项目是 Java，这一段最值得看。

### 场景一：新 Java 服务项目

推荐顺序：

1. 在项目根目录准备好 Git 仓库和构建文件，例如 `pom.xml` 或 `build.gradle`
2. 运行 `init`
3. 补齐 `docs/project/` 下项目级共享规范
4. 按团队要求调整 `.harness/spec-policy.json`
5. 审阅 `.harness/doc-impact-rules.json`，确保接口、数据库、安全、配置类变更的文档门禁符合团队预期
6. 新功能开始前生成功能级 spec
7. 人工改代码时先跑 `check-doc-impact`
8. 开发完成后做 `validate-spec`
9. 团队成熟后启用 `--strict`
10. 把 `check-doc-impact`、`lint-architecture`、`validate-spec --strict`、`check-doc-freshness` 接到 CI

示例：

```bash
bash scripts/init-harness.sh --project-name order-center --profile java-backend-service
bash scripts/new-feature-spec.sh --id FEAT-001 --title "订单查询" --owner alice --change-types api,db
bash scripts/check-doc-impact.sh --json --staged
bash scripts/validate-spec.sh --json
```

### 场景二：Java 批处理项目

建议初始化时直接指定：

```bash
bash scripts/init-harness.sh --project-name billing-job --profile java-batch-job
```

这样生成的项目级和功能级文档会更贴近批处理、任务调度、数据回填、失败重试等场景。

### 场景三：Java 适配层 / 网关 / 对接系统

建议初始化时直接指定：

```bash
bash scripts/init-harness.sh --project-name crm-adapter --profile java-adapter
```

这样更适合强调：

- 上下游系统对接
- 接口契约
- 幂等与补偿
- 兼容性和容错

### `profile` 应该怎么理解

`profile` 不是“代码生成器语言开关”，更像“模板画像”。

它表达的是：

- 这个项目大概率属于哪种工程形态
- 模板里的章节和关注点应该偏向什么

所以：

- 同样是 Java 项目，也可以用不同 profile
- profile 决定的是文档模板的默认视角，不是运行时行为

## 模板定制策略

这部分是后续部门推广时最关键的能力之一。

### 你有三种模板来源

| 层级 | 位置 | 适合谁 | 适合什么场景 |
|------|------|--------|--------------|
| 内置默认模板 | `assets/templates/` | Skill 维护者 | 部门级默认标准 |
| 项目级覆写模板 | `.harness/templates/` | 某个业务项目团队 | 这个项目的统一要求 |
| 用户级覆写模板 | `HARNESS_TEMPLATE_ROOT` 指向的目录 | 某个开发者 | 个人习惯或临时差异化 |

模板查找优先级：

1. `--template-root`
2. `HARNESS_TEMPLATE_ROOT`
3. `.harness/templates/`
4. `assets/templates/`

### 推荐的定制原则

如果你的诉求是部门统一标准：

- 改当前仓库的 `assets/templates/`

如果你的诉求是某个业务项目自己的固定要求：

- 改业务项目的 `.harness/templates/`

如果你的诉求只是个人习惯，不想影响团队：

- 使用 `HARNESS_TEMPLATE_ROOT`

### 一个常见误区

很多人会以为：

- 改了模板，历史上已经生成的文档会自动更新

当前不是这样的。

当前实现支持的是：

- 以后新生成的文档会应用新模板
- 历史文档是否需要同步，要通过 `check-template-drift` 审计出来
- 再由项目团队决定手工补齐或局部迁移

也就是说，当前已经支持“模板差异化”和“模板漂移识别”，但还没有做“自动迁移历史文档”。

## 模板升级与版本治理

这是后续推广时必须有的认知。

### 当前已经具备的能力

- 生成文档时会写入 `template_version`
- `.harness/spec-policy.json` 会记录模板包信息
- `.harness/doc-impact-rules.json` 会记录“哪些代码变更必须同步哪些文档”
- 可以检查文档元数据和模板包是否一致
- 可以基于 git diff 检查人工代码改动是否遗漏文档更新
- 可以检查 override 是否冗余、孤儿、或真实定制

### 当前还没有完全自动化的能力

- 还没有“一键把旧文档迁移到新模板”的自动迁移脚本

所以当前正确做法是：

1. 模板升级后，先更新模板版本和模板内容
2. 在业务项目里同步 `.harness/spec-policy.json`
3. 如有人工代码变更，先跑 `bash scripts/check-doc-impact.sh --json --staged` 或在 CI 中跑 base/head diff 检查
4. 跑 `bash scripts/check-template-drift.sh --json`
5. 识别出漂移文档、冗余 override、孤儿模板
6. 手工补齐或按项目需要重新整理
7. 最后跑 `bash scripts/validate-spec.sh --json --strict`

如果后面要继续完善，这里最值得新增的就是“模板迁移脚本”。

## 常见误区与常见问题

### 1. 为什么 `doc/` 用中文文件名，而 `docs/project/*.md` 还是英文路径

因为目标不同：

- `doc/` 是给维护者看的，中文命名更直观
- `docs/project/*.md` 和 `docs/features/*/*.md` 是给脚本、CI、工具链识别的，需要保持稳定

当前方案是：

- 文件内容默认中文
- 文件路径保持稳定

这是第一版更稳妥的做法。

### 2. 为什么新生成的 spec 不直接算“合格”

因为“生成出来”只代表结构搭好了，不代表内容已经经过团队确认。

所以这里故意分成两步：

- 第一步：生成结构
- 第二步：补齐内容后再通过严格校验

### 3. 什么时候用 `audit`，什么时候用 `check-doc-impact`，什么时候用 `validate-spec`，什么时候用 `check-template-drift`

可以这样记：

- `audit`：看项目整体 Harness 成熟度
- `check-doc-impact`：看本次代码 diff 是否要求同步更新文档
- `validate-spec`：看 spec 结构和内容是否达标
- `check-template-drift`：看模板版本、override 和历史文档是否脱节

### 4. `change_types` 能不能扩展

可以。

默认支持的是：

- `api`
- `db`
- `rollout`

如果团队还想支持别的文档，例如缓存变更、消息变更、SLA 变更，可以在 `.harness/spec-policy.json` 里继续扩展。

### 5. 项目级文档是不是都必须有

从推广和治理角度看，建议都保留。

最核心的通常是：

- 架构
- 开发规范
- 测试
- 安全
- 需求
- 接口总规范

如果项目团队想裁剪，也应该通过 `.harness/spec-policy.json` 明确裁剪规则，而不是口头约定。

## 维护这个仓库时，哪些文档需要同步更新

这部分是为了回答“`doc/` 下面还有哪些内容要跟着改”。

### 脚本行为变了

至少同步：

- `SKILL.md`
- `README.md`
- `doc/本地使用指南.md`

### `doc/` 的分层或文档职责变了

至少同步：

- `doc/文档导航.md`

### 模板版本、默认语言、默认 profile 变了

至少同步：

- `doc/本地使用指南.md`

### 方法论、推广策略、实践经验变了

优先同步：

- `doc/手册/Harness工程手册.md`
- `references/` 下相应参考文档

### 历史设计稿需要同步吗

通常不需要。

`doc/历史设计/` 和 `doc/归档/` 的定位就是“保留设计过程和旧方案”，不是当前实现真相。
