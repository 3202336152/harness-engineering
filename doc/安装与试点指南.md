# 安装与试点指南

这份文档只回答操作问题：

- 怎么在本地安装这个 Skill
- 怎么从远程发布源使用
- 安装后怎么触发
- 命令入口有哪些
- 怎么在别的业务项目里做首个试点

如果你想看的是 `/harness init`、`/harness audit`、`/harness plan` 的参数、产物、返回结果和示例，请直接看 [./命令使用说明.md](./命令使用说明.md)。

如果你关心的是“这个 Skill 本身有什么能力、解决什么问题、边界在哪里”，请看 [./能力与功能说明.md](./能力与功能说明.md)。

## 先选你的使用方式

先做一个判断：

- 想让本机多个项目都能用：全局安装
- 想只在某个业务项目里试点：项目级安装
- 想先验证脚本链路，不依赖 Skill 安装：直接脚本试点

## 一、如何安装

### 方案 A：从远程发布源安装

全局安装：

```bash
npx skills add 3202336152/harness-engineering -g -y
```

项目级安装：

```bash
cd /path/to/target-project
npx skills add 3202336152/harness-engineering -y
```

说明：

- `-g` 表示全局安装
- 不加 `-g` 时，默认安装到当前项目作用域
- `-y` 表示跳过确认
- 如果你只想装给某个 Agent，可以加 `-a <agent-name>`

### 方案 B：从本地仓库路径安装

如果当前仓库还没发布，或者你想先试本地改动版，推荐走“runtime-only 安装包”方式，不要直接把整个仓库根目录安装进去。

```bash
cd /path/to/harness-engineering
bash scripts/install-skill.sh --global
```

这个脚本内部会先执行：

- `bash scripts/export-skill-package.sh`
- 再把导出的 runtime-only 包安装到目标作用域

这样安装后的目录会只保留运行时需要的内容，例如：

- `SKILL.md`
- `scripts/` 下的核心运行脚本
- `scripts/lib/`
- `assets/templates/`
- `assets/hooks/`
- `assets/ci-templates/`
- `references/`
- `LICENSE`

这样安装后的目录不会再带：

- `tests/`
- `doc/`
- `README.md`
- `CHANGELOG.md`
- `scripts/export-skill-package.sh`
- `scripts/install-skill.sh`
- `scripts/publish-check.sh`
- `scripts/verify-spec-compliance.sh`

如果你想手工执行两步，也可以：

```bash
cd /path/to/harness-engineering
bash scripts/export-skill-package.sh --output-dir .build/skill-package
npx skills add .build/skill-package/harness-engineering -g -y
```

如果你当前就在本仓库根目录，可以先检查导出的本地包是否可识别：

```bash
cd /path/to/harness-engineering
npx skills add .build/skill-package/harness-engineering -l
```

### 方案 C：不安装 Skill，直接调用脚本

这是最稳、最适合做 PoC 的方式。

```bash
HARNESS_SKILL_DIR=/path/to/harness-engineering
cd /path/to/target-project

bash "$HARNESS_SKILL_DIR/scripts/init-harness.sh" --project-name order-center
```

补充说明：

- 安装包里保留 `assets/hooks/` 和 `assets/ci-templates/`，是为了让安装后的 Skill 也能生成本地 hook 和 CI 门禁文件
- 也就是说，别的用户安装后执行 `init --with-git-hook`、`init --with-husky`、`init --with-github-actions` 时，不需要再依赖这个仓库的完整源码

## 二、如何确认安装成功

```bash
npx skills ls -g --json
npx skills ls --json
```

重点看输出里的：

- `name`
- `path`
- `scope`

只要能看到 `harness-engineering`，并且 `path` 指向预期目录，就说明安装成功。

## 三、安装后怎么触发

当前建议把触发方式理解成三层。

### 1. Slash 命令触发

适合支持 slash command 的 Agent。

| 输入 | 作用 |
|------|------|
| `/harness init` | 初始化 Harness 结构 |
| `/harness audit` | 审计 Harness 成熟度 |
| `/harness plan <任务>` | 生成结构化执行计划 |

注意：

- 不是所有 Agent 都支持 slash command
- 如果当前 Agent 不支持，直接用自然语言触发或终端脚本触发即可

### 2. 自然语言触发

常见说法：

- “帮我初始化当前项目的 Harness 结构”
- “检查一下这个项目的 Harness 成熟度”
- “按 Harness 规范帮我规划这个功能的实现”
- “给 FEAT-001 生成功能 spec 并跑一遍 verify”
- “上下文压缩了，帮我恢复上一次的 Harness 任务状态”

### 3. 终端脚本触发

这是最稳定、最适合试点和 CI 的方式。

```bash
bash /path/to/harness-engineering/scripts/init-harness.sh --project-name order-center
bash /path/to/harness-engineering/scripts/harness-exec.sh verify --feature-id FEAT-001 --json
bash /path/to/harness-engineering/scripts/harness-exec.sh restore --feature-id FEAT-001 --json
```

如果平台支持自动化或定时任务，建议再补一条“每周执行一次 `bash /path/to/harness-engineering/scripts/audit-harness.sh`”的自动化任务，让 `.harness/runtime/last-audit.json` 保持新鲜。

如果你希望目标项目不是“只生成规范骨架”，而是直接具备真实门禁，推荐在初始化时显式打开约束：

```bash
bash /path/to/harness-engineering/scripts/init-harness.sh --project-name order-center --with-strong-constraints
```

这会额外生成：

- `.harness/skill-runtime/harness-engineering/`
- `.git/hooks/pre-commit`
- `.github/workflows/harness-guardrails.yml`

对于个人 Java 项目，推荐把这条命令当作默认起点，因为它会同时打开：

- 入口文档里的强约束提示
- 提交前的 doc-impact + strict spec 校验
- 合并前的 CI 校验

这样质量约束更多依赖机械化门禁，而不是依赖“记得手动运行 `/harness plan` 或校验脚本”。

如果你更偏向 Husky，可以改用：

```bash
bash /path/to/harness-engineering/scripts/init-harness.sh --project-name order-center --with-husky
```

这会生成 `.husky/pre-commit`，并自动把 `core.hooksPath` 指向 `.husky`

如果你希望 Husky 也在本地直接阻断 placeholder spec，可以再加上：

```bash
bash /path/to/harness-engineering/scripts/init-harness.sh --project-name order-center --with-husky --with-strict-spec-checks
```

初始化完成后，还建议立刻进入“第二阶段补全文档”：

- 不要把 `init` 理解成“已经自动读完项目并写好文档”，它生成的是结构化骨架
- 对 Java 项目，推荐先运行 `bash /path/to/harness-engineering/scripts/scan-java-project.sh --json`，把 `.harness/runtime/java-doc-scan.json` 作为全量扫描清单
- 更合理的做法是让宿主模型先核对扫描清单，再做关键文件深读，最后补全 `docs/project/`
- 推荐关键深读清单：`pom.xml` / `build.gradle*`、启动类或主入口、`src/main/java` 前两层包结构、主要 `Controller` / `Facade` / `Listener` / `Job`、核心 `ApplicationService` / `DomainService`、`application.yml` / `application-*.yml`
- 不要求把所有源码全文一次性塞进上下文，但至少要先完成全量扫描，再覆盖代表性入口、核心链路、关键配置和主要外部集成
- 如果有没覆盖到的模块，在文档里写“待确认 / 未覆盖范围”；补全后执行 `bash /path/to/harness-engineering/scripts/validate-spec.sh --json --strict`
- 对 Java profile，`validate-spec --strict` 会检查扫描清单中的模块、入口、服务、外部依赖是否真的在项目文档里出现

## 四、当前有哪些命令入口

### 1. 对话层入口

| 命令 | 作用 |
|------|------|
| `/harness init` | 初始化项目 Harness 结构 |
| `/harness audit` | 审计成熟度和缺口 |
| `/harness plan` | 生成结构化执行计划 |

### 2. 脚本层入口

| 类别 | 命令 | 作用 |
|------|------|------|
| 初始化 | `bash scripts/init-harness.sh` | 初始化项目结构和 `.harness/` 配置 |
| 审计 | `bash scripts/audit-harness.sh` | 输出成熟度评分 |
| 规划 | `bash scripts/plan-harness.sh` | 输出 `md + json` 执行计划 |
| 功能 spec | `bash scripts/new-feature-spec.sh` | 生成功能级 spec 包 |
| 上下文 | `bash scripts/resolve-task-context.sh` | 生成任务上下文 bundle |
| 文档门禁 | `bash scripts/check-doc-impact.sh` | 检查代码变更对应的文档更新要求 |
| 结构校验 | `bash scripts/validate-spec.sh` | 校验 spec 结构与质量 |
| 回滚校验 | `bash scripts/check-rollback-readiness.sh` | 校验高风险功能的回滚准备度 |
| 自治入口 | `bash scripts/harness-exec.sh prepare` | 一次生成 spec、计划、上下文 |
| 自治入口 | `bash scripts/harness-exec.sh verify` | 聚合校验并落账本、记忆、证据 |
| 自治入口 | `bash scripts/harness-exec.sh run` | 串联 prepare、verify、safe autofix、reverify |
| 自治恢复 | `bash scripts/harness-exec.sh restore` | 从 runtime 记忆、进度和功能状态中恢复最近任务上下文 |
| 模板治理 | `bash scripts/check-template-drift.sh` | 检查模板漂移和 override 卫生 |
| 模板迁移 | `bash scripts/migrate-template-docs.sh` | 带备份迁移历史文档 |
| 证据采集 | `bash scripts/collect-runtime-evidence.sh` | 采集命令输出与文件证据 |
| 运行清理 | `bash scripts/harness-gc.sh` | 清理旧 runtime 产物 |

## 五、在其他项目中如何试点

最推荐走两条路线。

### 路线 1：先直接脚本试点

适合：

- 想快速验证
- 不依赖 Agent 的命令能力
- 更关心产物和链路是否真实可落地

步骤：

1. 选一个已经有 Git 管理的业务项目
2. 进入目标项目
3. 直接调用当前仓库脚本
4. 生成项目级规范和首个功能 spec
5. 实现代码后跑 `verify`

示例：

```bash
HARNESS_SKILL_DIR=/path/to/harness-engineering
cd /path/to/target-project

bash "$HARNESS_SKILL_DIR/scripts/init-harness.sh" --project-name order-center
bash "$HARNESS_SKILL_DIR/scripts/new-feature-spec.sh" --id FEAT-001 --title "订单查询" --owner alice --change-types api,rollout
bash "$HARNESS_SKILL_DIR/scripts/harness-exec.sh" prepare --task "新增订单查询能力" --feature-id FEAT-001 --title "订单查询" --owner alice --change-types api,rollout --json
# 在这里实现代码与补充文档
bash "$HARNESS_SKILL_DIR/scripts/harness-exec.sh" verify --feature-id FEAT-001 --json
bash "$HARNESS_SKILL_DIR/scripts/harness-exec.sh" restore --feature-id FEAT-001 --json
```

### 路线 2：安装为 Skill 后在业务项目里触发

适合：

- 你已经确认脚本可用
- 你想验证 Agent 层体验
- 你想在多个业务项目里复用

推荐步骤：

1. 先全局安装或项目级安装 Skill
2. 打开目标业务项目
3. 先执行 `/harness init` 或自然语言初始化
4. 再生成首个功能 spec 和执行计划
5. 开发完成后运行 `verify` 或 `run`

推荐对话示例：

- `/harness init`
- `/harness audit`
- `/harness plan 新增订单查询能力`
- “请按 Harness 规范为 FEAT-001 初始化功能 spec，并生成执行计划和上下文 bundle”
- “请对 FEAT-001 跑一遍 Harness verify，并告诉我 evidence 在哪里”

## 六、试点成功最少要看到什么

你至少要看到下面这些东西真正落到目标项目里：

- `docs/project/项目架构.md`
- `docs/project/开发规范.md`
- `docs/project/测试策略.md`
- `docs/features/FEAT-001-*/`
- `.harness/spec-policy.json`
- `.harness/runs/ledger.jsonl`
- `.harness/metrics/ledger.jsonl`
- `.harness/runtime/task-memory.json`
- `.harness/runtime/progress.md`
- `.harness/evidence/<run-id>/manifest.json`

如果你启用了强约束模式，还应该额外看到：

- `.harness/skill-runtime/harness-engineering/`
- `.git/hooks/pre-commit` 或 `.husky/pre-commit`
- `.github/workflows/harness-guardrails.yml`

如果这些产物都已经稳定出现，说明这套 Skill 已经具备了“基础可试点”的条件。
