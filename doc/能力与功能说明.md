# 能力与功能说明

这份文档只回答一件事：

这个 `harness-engineering` Skill 当前到底提供了什么能力，解决什么问题，产出什么东西，边界又在哪里。

如果你关心的是“怎么安装、怎么触发、怎么在别的项目试点”，请看 [./安装与试点指南.md](./安装与试点指南.md)。

## 一句话定位

这个仓库不是业务项目本身，而是一个“Skill + 模板 + 脚本”的标准化工具包。

它的目标是给业务项目补齐一套可被 Agent 和 CI 机械执行的 Harness：

- 入口文档
- 项目级规范
- 功能级 spec
- 上下文策略
- 运行策略
- 校验门禁
- 反馈闭环
- 模板升级治理

## 它主要解决什么问题

对应常见的 spec 治理问题，当前已经有下面这些机械化能力。

| 问题 | 当前对应能力 |
|------|--------------|
| 依赖人工遵守 | `check-doc-impact`、`validate-spec`、`lint-architecture`、`check-rollback-readiness` 把规则变成脚本门禁 |
| 上下文管理混乱 | `.harness/context-policy.json` + `resolve-task-context.sh` 输出机器可读上下文 bundle |
| 文档熵增 | 模板元数据、strict 校验、template drift 检查、历史文档迁移、safe autofix 控制结构漂移 |
| 缺乏反馈闭环 | `harness-exec.sh verify/run` 聚合校验，并沉淀 run record、metrics、memory、progress、evidence |
| 会话切换后上下文丢失 | `harness-exec.sh restore` 从 runtime 记忆、进度和功能状态中恢复最近任务摘要 |

## 当前有哪些核心能力

### 1. Harness 初始化

通过 `init-harness.sh` 或 `/harness init`，可以在业务项目里初始化：

- `AGENTS.md`
- `CLAUDE.md`
- `docs/project/*.md`
- `docs/project/项目架构.md`
- `docs/project/开发规范.md`
- `docs/project/测试策略.md`
- `docs/features/`
- `docs/exec-plans/`
- `.harness/*.json`
- `.harness/runtime/`
- `.harness/evidence/`
- `.harness/metrics/`

初始化后，项目就具备了规范、模板、门禁配置和基础自治入口。

### 2. Harness 成熟度审计

通过 `audit-harness.sh` 或 `/harness audit`，可以给当前项目做成熟度评分。

它会看：

- 入口文档
- 文档结构
- 文档新鲜度
- 架构边界
- 测试准备度
- 自动化
- 执行计划
- 安全治理

### 3. 执行计划与上下文收敛

当前已经支持把“做什么”和“读什么”拆开治理。

相关能力：

- `plan-harness.sh`：输出 Markdown 执行计划和 JSON 机器计划
- `resolve-task-context.sh`：按规则生成任务上下文 bundle
- `harness-exec.sh prepare`：一次串起 spec、计划、上下文
- `harness-exec.sh restore`：在上下文压缩后恢复最近任务和推荐上下文

### 4. 项目级 spec 与功能级 spec 工作流

项目级规范统一放在：

```text
docs/project/
```

功能级 spec 统一放在：

```text
docs/features/<feature-id>-<title-slug>/
```

当前默认生成的 `md` 文件名也是中文，例如：

- `docs/project/项目架构.md`
- `docs/project/接口规范.md`
- `docs/features/FEAT-001-order-query/功能概览.md`
- `docs/features/FEAT-001-order-query/接口设计.md`

当前功能级 spec 支持按 `change_types` 自动扩展，例如：

- `api`
- `db`
- `rollout`

并自动生成 `manifest.json`，把：

- 必需文档
- 相关项目文档
- 验证步骤
- 风险等级
- 回滚要求
- 模板元数据

一起落成机器可读产物。

### 5. 规则门禁与文档治理

当前已经内建几类核心门禁。

| 脚本 | 作用 |
|------|------|
| `check-doc-impact.sh` | 检查代码变更是否要求同步更新文档 |
| `validate-spec.sh` | 检查 spec 结构和内容完整性 |
| `validate-spec.sh --strict` | 把占位内容、缺少章节等问题升级为失败 |
| `check-template-drift.sh` | 检查模板元数据漂移、冗余 override、孤儿 override |
| `check-rollback-readiness.sh` | 检查高风险功能的回滚准备度 |
| `lint-architecture.sh` | 检查架构层级边界 |
| `check-doc-freshness.sh` | 检查文档新鲜度 |

### 6. 历史文档迁移

现在已经补上了历史模板文档迁移能力：

```bash
bash scripts/migrate-template-docs.sh --json
```

它的行为不是“粗暴重写所有文档”，而是：

1. 先识别受影响文档
2. 先备份到 `.harness/migrations/<timestamp>/backup/`
3. 再执行 safe autofix 级别的结构迁移
4. 输出 `report.json`

### 7. 运行闭环、账本、记忆和证据

通过：

```bash
bash scripts/harness-exec.sh verify --feature-id FEAT-001 --json
bash scripts/harness-exec.sh run --task "..." --feature-id FEAT-001 --title "..." --json
bash scripts/harness-exec.sh restore --feature-id FEAT-001 --json
```

当前会自动沉淀：

- `.harness/runs/<run-id>.json`
- `.harness/runs/ledger.jsonl`
- `.harness/metrics/ledger.jsonl`
- `.harness/runtime/task-memory.json`
- `.harness/runtime/progress.md`
- `.harness/evidence/<run-id>/`

并且可以把这些运行期产物重新还原成可读的恢复结果，包括：

- 最近一次任务和 feature
- 最近一次执行模式与状态
- `状态.md` 中尚未完成的检查项
- 推荐重新注入给 Agent 的上下文文件列表

### 8. 可观测性接入

项目可以通过：

```text
.harness/observability-policy.json
```

配置：

- 总是采集的命令
- 运行期按需采集的命令
- 需要复制沉淀的文件产物

然后由 `collect-runtime-evidence.sh` 或 `harness-exec verify/run` 自动采集到证据目录。

### 9. Runtime 保留清理

当前可以通过：

```bash
bash scripts/harness-gc.sh --json
```

按 `.harness/run-policy.json` 的保留策略清理旧上下文、旧 run record、旧 evidence。

## 当前能力边界

当前已经达到“基础可试点”的标准，但还没有达到“完全自治平台”的标准。

### 已经具备的

- 初始化结构
- 规则门禁
- 上下文 bundle
- 功能 spec 工作流
- 模板漂移治理
- 历史文档安全迁移
- run ledger / metrics ledger
- task memory / progress
- evidence deposition
- retention GC

### 还没有完全做到的

- 不会自动理解业务语义并重写历史文档全文
- 不会自动修改业务代码来满足规范
- 没有内建远端 dashboard、远端观测平台或集中式指标平台
- `autofix-safe` 主要还是修结构和元数据，不修业务语义

## 更适合拿去试点的场景

- 已经有 Git 管理的业务项目
- 团队愿意先从“规范 + 门禁 + 证据”开始
- 希望先把项目级规范、功能级 spec 和运行闭环稳定下来
