# 命令使用说明

这份文档只介绍三个核心命令的具体用法：

- `/harness init`
- `/harness audit`
- `/harness plan`

如果你关心的是“怎么安装、怎么在别的项目试点、怎么远程使用”，请看 [./安装与试点指南.md](./安装与试点指南.md)。
如果你关心的是“这个 Skill 到底提供什么能力、边界在哪里”，请看 [./能力与功能说明.md](./能力与功能说明.md)。

## 先看总览

| 命令 | 适用阶段 | 解决的问题 | 脚本等价调用 |
|------|----------|------------|--------------|
| `/harness init` | 项目接入初期 | 给项目补齐 Harness 基础结构、核心规范文档和 `.harness/` 配置 | `bash scripts/init-harness.sh` |
| `/harness audit` | 接入前评估、治理巡检 | 判断当前项目的 Harness 成熟度，定位主要缺口 | `bash scripts/audit-harness.sh` |
| `/harness plan` | 某个功能或任务开始前 | 为具体变更生成结构化执行计划和机器可读计划 JSON | `bash scripts/plan-harness.sh --task "<任务>"` |

建议的使用顺序通常是：

1. 先用 `/harness init` 把基础结构补齐
2. 再用 `/harness audit` 看当前成熟度和短板
3. 具体做某个功能时，再用 `/harness plan`

## 使用前提

- 你需要已经安装好 `harness-engineering` Skill，或者直接从本仓库调用脚本
- 当前终端目录最好就是目标项目根目录
- 如果当前 Agent 不支持 slash command，就直接使用自然语言或脚本调用
- 如果你需要显式传参数，例如 `--profile`、`--dry-run`、`--output-dir`，优先使用脚本调用

## `/harness init`

### 作用

`/harness init` 用来给一个项目建立最小可用的 Harness 基础设施。它不是只建几个目录，而是同时补齐：

- 入口文档
- 项目级规范文档
- 执行计划目录
- `.harness/` 下的机器可读策略文件
- runtime、evidence、metrics 等后续自治能力要依赖的目录

### 适用场景

- 新项目刚开始接入 AI 编码代理
- 老项目还没有 `AGENTS.md`、`docs/project/`、`.harness/`
- 你想先初始化一套标准结构，再按项目实际补内容

### 触发方式

Slash 命令：

```text
/harness init
```

自然语言：

- “帮我初始化当前项目的 Harness 结构”
- “给这个仓库补齐 Harness 基础文档和 `.harness` 配置”

脚本调用：

```bash
bash scripts/init-harness.sh [--project-name <name>] [--description <text>] [--template-root <path>] [--profile <name>] [--with-git-hook] [--with-husky] [--with-github-actions] [--with-strong-constraints] [--force] [--dry-run]
```

### 参数说明

| 参数 | 是否必填 | 说明 |
|------|----------|------|
| `--project-name <name>` | 否 | 指定项目名；默认使用当前目录名 |
| `--description <text>` | 否 | 写入模板中的项目说明；默认是占位文本 |
| `--template-root <path>` | 否 | 指定本次初始化使用的模板根目录 |
| `--profile <name>` | 否 | 指定模板画像；未指定时会根据项目技术栈自动推断 |
| `--with-git-hook` | 否 | 生成 `.git/hooks/pre-commit`，把文档门禁接入本地提交 |
| `--with-husky` | 否 | 生成 `.husky/pre-commit`，并把仓库 `core.hooksPath` 指到 `.husky` |
| `--with-github-actions` | 否 | 生成 `.github/workflows/harness-guardrails.yml`，把校验接入 GitHub Actions |
| `--with-strong-constraints` | 否 | 组合开启 `--with-git-hook` 与 `--with-github-actions` |
| `--force` | 否 | 强制覆盖已存在文件；默认只创建缺失文件 |
| `--dry-run` | 否 | 只输出将要创建/跳过的内容，不实际写文件 |

模板查找优先级如下：

1. `--template-root <path>`
2. 环境变量 `HARNESS_TEMPLATE_ROOT`
3. 项目内 `.harness/templates/`
4. 内置 `assets/templates/`

### 它会做什么

执行时会按当前实现完成下面这些动作：

1. 检查当前目录是否在 Git 仓库内
2. 识别项目技术栈
3. 创建标准目录结构
4. 渲染文档模板和 `.harness/*.json` 策略文件
5. 如果显式开启了约束选项，则 vendoring Skill runtime 到 `.harness/skill-runtime/harness-engineering`
6. 按选项生成 `.git/hooks/pre-commit`、`.husky/pre-commit`、`.github/workflows/harness-guardrails.yml`
7. 对已存在文件默认跳过，不覆盖
8. 输出一份 JSON 摘要，告诉你创建了什么、跳过了什么、接下来该做什么

当前支持的技术栈识别包括：

- Node：`package.json`
- Java Maven：`pom.xml`
- Java Gradle：`build.gradle` / `build.gradle.kts`
- Python：`pyproject.toml` / `setup.py`
- Go：`go.mod`
- Rust：`Cargo.toml`

### 执行后会生成什么

典型目录：

- `docs/project/`
- `docs/features/`
- `docs/exec-plans/active/`
- `docs/exec-plans/completed/`
- `docs/exec-plans/tech-debt/`
- `.harness/runtime/`
- `.harness/runs/`
- `.harness/evidence/`
- `.harness/metrics/`

启用约束时的额外产物：

- `.harness/skill-runtime/harness-engineering/`
- `.git/hooks/pre-commit`
- `.husky/pre-commit`
- `.github/workflows/harness-guardrails.yml`

典型文档：

- `AGENTS.md`
- `CLAUDE.md`
- `docs/project/核心信念.md`
- `docs/project/项目架构.md`
- `docs/project/项目设计.md`
- `docs/project/接口规范.md`
- `docs/project/开发规范.md`
- `docs/project/需求说明.md`
- `docs/project/测试策略.md`
- `docs/project/安全规范.md`
- `docs/project/运行基线.md`
- `docs/project/可观测性基线.md`

典型策略文件：

- `.harness/architecture.json`
- `.harness/spec-policy.json`
- `.harness/doc-impact-rules.json`
- `.harness/context-policy.json`
- `.harness/run-policy.json`
- `.harness/observability-policy.json`
- `.harness/runtime/task-memory.json`
- `.harness/runtime/progress.md`

### 返回结果怎么看

脚本最终会输出 JSON，核心字段包括：

- `status`：执行状态
- `project`：项目名
- `created_files`：本次创建的文件
- `created_dirs`：本次创建的目录
- `skipped_files`：因已存在而跳过的文件
- `enabled_guardrails`：本次启用的约束项
- `detected_stack`：识别出的技术栈
- `next_steps`：后续建议动作

你最应该先看的是：

1. `skipped_files`
2. `detected_stack`
3. `next_steps`

这样能快速知道是不是用了你期望的画像、是不是有文件没被覆盖、下一步应该补哪些内容。

### 常见示例

初始化当前项目：

```bash
bash scripts/init-harness.sh
```

指定项目名和描述：

```bash
bash scripts/init-harness.sh --project-name order-center --description "订单中心服务"
```

先预览不落盘：

```bash
bash scripts/init-harness.sh --dry-run
```

初始化并直接接入本地 hook + GitHub Actions：

```bash
bash scripts/init-harness.sh --with-strong-constraints
```

初始化并接入 Husky：

```bash
bash scripts/init-harness.sh --with-husky
```

强制按新模板覆盖已有文件：

```bash
bash scripts/init-harness.sh --force
```

### 注意事项

- 默认不会覆盖已有文件，所以已经手工维护过的文档不会被自动改掉
- 如果你希望升级模板并迁移历史文档，不要只依赖 `init --force`，更适合用 `scripts/migrate-template-docs.sh`
- `init` 只是把“基础骨架”搭起来，项目级文档内容仍然需要结合真实业务补全
- 默认不会主动改你的提交链路；只有显式传 `--with-git-hook`、`--with-husky`、`--with-github-actions` 时才会生成真实门禁
- 如果你启用了 GitHub Actions 门禁，需要把 `.harness/skill-runtime/harness-engineering` 一起提交到仓库，CI 才能稳定复用同一套 runtime

## `/harness audit`

### 作用

`/harness audit` 用来评估当前项目的 Harness 成熟度。它不会改文件，只会输出一份机器可读的审计结果，告诉你当前在哪个成熟度等级，以及最优先该补什么。

### 适用场景

- 你想判断一个仓库是否已经达到试点标准
- 你想知道离 “Autonomous Harness” 还差哪些基础能力
- 你想在接入前先做一次治理盘点

### 触发方式

Slash 命令：

```text
/harness audit
```

自然语言：

- “审计一下这个项目的 Harness 成熟度”
- “看看当前仓库离自治 Harness 还有哪些缺口”

脚本调用：

```bash
bash scripts/audit-harness.sh
```

### 参数说明

当前实现没有额外参数。命令会直接基于当前项目目录做检查。

### 它检查什么

当前审计会评估 8 个维度：

1. 入口文档
2. 文档结构
3. 文档新鲜度
4. 架构约束
5. 测试准备度
6. 自动化
7. 执行计划
8. 安全治理

当前权重如下：

- 入口文档：15%
- 文档结构：15%
- 文档新鲜度：10%
- 架构约束：15%
- 测试准备度：15%
- 自动化：10%
- 执行计划：10%
- 安全治理：10%

### 分数和等级怎么对应

| 总分 | 等级 | 含义 |
|------|------|------|
| `0-24` | Level 0 | No Harness |
| `25-49` | Level 1 | Basic Harness |
| `50-69` | Level 2 | Constrained Harness |
| `70-89` | Level 3 | Observable Harness |
| `90-100` | Level 4 | Autonomous Harness |

### 返回结果怎么看

这个命令只会往标准输出打印 JSON，适合给 Agent、CI、脚本继续消费。

核心字段包括：

- `status`：通常为 `completed`
- `overall_score`：总分
- `maturity_level`：等级数字
- `maturity_label`：等级名称
- `dimensions`：各维度详细评分
- `priority_fixes`：优先修复项

你可以按这个顺序看结果：

1. 先看 `overall_score` 和 `maturity_label`
2. 再看 `priority_fixes`
3. 最后逐项看 `dimensions`

`priority_fixes` 很适合直接转成你的治理待办，因为它已经按低分维度给出了优先级和修复建议。

### 常见示例

直接执行：

```bash
bash scripts/audit-harness.sh
```

如果你本地装了 `jq`，更适合格式化查看：

```bash
bash scripts/audit-harness.sh | jq .
```

### 注意事项

- 这个命令不会自动修复问题，只负责评估和给出建议
- 文档新鲜度依赖 Git 历史；如果当前目录不是 Git 仓库，对这一项的判断会弱一些
- `audit` 的结果是“当前状态快照”，适合在初始化后、试点前、接入 CI 前各跑一次

## `/harness plan`

### 作用

`/harness plan` 用来给“某一个具体任务”生成结构化执行计划。它会同时生成：

- 一份给人看的 Markdown 执行计划
- 一份给脚本或 Agent 用的 JSON 计划

它的目标不是创建整个项目骨架，而是让某次变更在开始前就对齐项目级规范、验证要求和风险级别。

### 适用场景

- 你准备开始一个新功能或改造任务
- 你希望 Agent 在动手前先把目标、约束、验收和分阶段步骤写清楚
- 你希望后续自动化能读取计划的风险级别、所需检查项和回滚要求

### 触发方式

Slash 命令：

```text
/harness plan 新增订单查询能力
```

自然语言：

- “按 Harness 规范给我规划一下订单查询功能的实现”
- “为 FEAT-001 生成执行计划，要求带风险等级和校验项”

脚本调用：

```bash
bash scripts/plan-harness.sh --task "<任务描述>" --agent "<agent-name>" [--feature-id <id>] [--change-types <csv>] [--output-dir <path>] [--dry-run]
```

### 参数说明

| 参数 | 是否必填 | 说明 |
|------|----------|------|
| `--task <description>` | 是 | 任务描述；也支持 `--title` 作为别名 |
| `--agent <name>` | 否 | 记录生成计划的 Agent 名称；默认 `unknown-agent` |
| `--feature-id <id>` | 否 | 关联功能编号，例如 `FEAT-001` |
| `--change-types <csv>` | 否 | 变更类型，逗号分隔，例如 `api,db,rollout` |
| `--output-dir <path>` | 否 | 输出目录；默认 `docs/exec-plans/active` |
| `--dry-run` | 否 | 预演模式，不真正写文件 |

### 它会做什么

当前实现的关键行为是：

1. 校验是否传入任务描述
2. 把任务标题转成文件名 slug
3. 在输出目录下写入 `md + json`
4. 自动引用项目级规范作为约束
5. 根据 `change-types` 推断风险等级和是否要求回滚

风险等级规则当前是：

- 包含 `db` 或 `rollout`：`high`
- 含 `api` 但不含前两者：`medium`
- 其他：`low`

回滚要求规则当前是：

- 包含 `db` 或 `rollout`：`rollback_required = true`
- 其他：`rollback_required = false`

### 执行后会生成什么

默认输出到：

```text
docs/exec-plans/active/
```

例如任务是“新增订单查询能力”，可能生成：

- `docs/exec-plans/active/新增订单查询能力.md`
- `docs/exec-plans/active/新增订单查询能力.json`

Markdown 计划里会包含：

- 状态
- 创建时间
- Agent
- Objective
- Constraints
- Acceptance Criteria
- Implementation Steps
- Decision Log
- Issues

JSON 计划里会包含：

- `task`
- `feature_id`
- `change_types`
- `agent`
- `created`
- `markdown_path`
- `required_docs`
- `required_checks`
- `risk_level`
- `rollback_required`

### 返回结果怎么看

命令本身会输出一份 JSON 摘要，核心字段包括：

- `status`
- `title`
- `path`
- `machine_plan_path`
- `agent`
- `dry_run`
- `references`

你最应该先看：

1. `path` 和 `machine_plan_path` 是否落在预期目录
2. `references` 是否引用到了正确的项目级规范
3. 生成的 JSON 里 `risk_level` 和 `rollback_required` 是否符合任务实际风险

### 常见示例

为一个任务生成默认计划：

```bash
bash scripts/plan-harness.sh --task "新增订单查询能力" --agent codex
```

带功能编号和变更类型：

```bash
bash scripts/plan-harness.sh \
  --task "新增订单查询能力" \
  --agent codex \
  --feature-id FEAT-001 \
  --change-types api,rollout
```

输出到自定义目录：

```bash
bash scripts/plan-harness.sh \
  --task "重构库存同步链路" \
  --agent codex \
  --output-dir docs/exec-plans/tech-debt
```

只预览不写文件：

```bash
bash scripts/plan-harness.sh --task "新增订单查询能力" --dry-run
```

### 注意事项

- `plan` 不会自动创建功能 spec；如果你希望把 spec、计划、上下文 bundle 一次准备好，更适合用 `scripts/harness-exec.sh prepare`
- `plan` 也不会自动执行校验；开发完成后还需要再跑 `verify` 或 `run`
- 如果你没有传 `--task`，脚本会输出错误 JSON 并退出

## 三个命令之间的边界

为了避免用混，最好把它们理解成三个不同层级的动作：

- `/harness init`：给项目建立基础设施
- `/harness audit`：给项目做体检打分
- `/harness plan`：给某个具体任务做执行规划

也就是说：

- `init` 面向“项目”
- `audit` 面向“项目当前状态”
- `plan` 面向“单个变更任务”

## 推荐的最小落地路径

如果你要在一个业务项目里第一次试用，推荐这样走：

1. 进入目标项目根目录
2. 执行 `init`
3. 补齐最关键的项目级规范
4. 执行 `audit`
5. 选择一个试点功能，执行 `plan`
6. 实现代码后，再接 `harness-exec.sh verify` 或 `harness-exec.sh run`

这样更容易把“初始化”“治理评估”“具体任务执行”三件事拆开看，不会把命令职责混在一起。

补充：

- 如果你在开发过程中发生了上下文压缩或会话切换，可以用 `bash scripts/harness-exec.sh restore --feature-id FEAT-001 --json` 恢复最近一次任务状态、待办检查项和推荐上下文文件
